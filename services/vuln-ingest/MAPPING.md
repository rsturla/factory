# Mapping RPMs to Vulnerabilities

How downstream services should use vuln-ingest to determine which CVEs affect Hummingbird RPM packages.

## The Problem

Upstream CVE data uses inconsistent identifiers for the same software:

| Source | How it identifies libxml2 |
|--------|--------------------------|
| NVD | `vendor=xmlsoft`, `package_name=libxml2` |
| cvelistV5 | `vendor=xmlsoft`, `package_name=libxml2` |
| GHSA/OSV | `ecosystem=OSS-Fuzz`, `package_name=libxml2` |
| Kernel CVEs | `ecosystem=Linux`, `package_name=linux` |

Meanwhile, the Hummingbird RPM is `libxml2-2.12.5-3.hb1.x86_64.rpm`. No source knows this name.

## The Solution

Three pieces work together:

```
rpms monorepo          vuln-ingest             matching service
(identity mapping)  +  (CVE knowledge base)  = (automated matching)
```

### 1. RPM Identity Mapping (rpms monorepo)

Each RPM package declares its upstream CVE identifiers in its existing metadata file:

```json
// rpms/libxml2/metadata/libxml2.json
{
  "name": "libxml2",
  "upstream_version": "2.12.5",
  "cve_identifiers": {
    "vendor": "xmlsoft",
    "product": "libxml2",
    "purl": "pkg:generic/libxml2"
  }
}
```

The `cve_identifiers` block is the only addition needed — the rest of the metadata file already exists. The `vendor` + `product` fields map to how NVD and cvelistV5 identify the project. The `purl` field maps to how OSV/GHSA identify it. The `product` field doubles as the `package_name` used across all sources.

`upstream_version` should already be present or extractable from the spec `Version:` field. The matching service reads it to compare against affected version ranges.

For packages that bundle dependencies (e.g., Go binaries), the SBOM generated by Konflux provides the identity mapping automatically via PURLs. The `cve_identifiers` block is only needed for the top-level upstream project.

### 2. Query Strategy

The matching service queries vuln-ingest using **all identifiers** for each RPM and unions the results. No single identifier scheme has complete coverage.

```
For each RPM:
  1. Query by vendor + product (catches NVD + cvelistV5 data)
     POST /v1/affected:batchQuery
     {"queries": [{"vendor": "xmlsoft", "package_name": "libxml2"}]}

  2. Query by package_name alone (catches cross-ecosystem matches)
     Already included in batch query — vendor+product query also 
     matches package_name-only records from other sources

  3. Query by purl (catches SBOM-linked advisories)
     {"queries": [{"purl": "pkg:generic/libxml2"}]}
```

Use `POST /v1/affected:batchQuery` to query all 440 RPMs in a single HTTP call (max 500 queries per request).

### 3. Version Comparison

For each matched CVE, compare the RPM's upstream version against the affected version ranges:

```
CVE-2024-1234:
  affected_packages:
    - vendor: xmlsoft
      package_name: libxml2
      version_ranges:
        - range_type: ""           # NVD — no explicit type
          introduced: "2.9.0"
          fixed: "2.12.6"
      source: nvd

RPM upstream_version: 2.12.5

2.9.0 ≤ 2.12.5 < 2.12.6 → AFFECTED
```

**Use `range_type` to select the comparator:**

| range_type | Comparator | Used by |
|------------|-----------|---------|
| `SEMVER` | Semantic versioning | OSV, GHSA, kernel |
| `ECOSYSTEM` | Ecosystem-native (PyPI PEP 440, npm semver) | OSV |
| `GIT` | Not comparable — commit hashes, skip | OSV, kernel |
| `semver` | Semantic versioning | cvelistV5 |
| `git` | Not comparable — commit hashes, skip | cvelistV5 |
| `custom` | Vendor-specific, flag for manual review | cvelistV5 |
| empty | Assume semver-like for C/system libraries | NVD |

**Handle `introduced_exclusive`:** When true (NVD `versionStartExcluding`), the boundary is `>` not `>=`.

**Skip `GIT`/`git` ranges:** These contain commit hashes, not version strings. They're useful for source-level analysis but not for RPM version matching.

### 4. Quality Flags

Not all data is reliable. The matching service should handle quality flags:

| Flag | Meaning | Matching action |
|------|---------|----------------|
| (none) | Clean data | Automated match, high confidence |
| `no_upper_bound` | Affected range has no fix version | Assume affected, flag for review |
| `empty_range` | No version data at all | Cannot match automatically, queue for triage |
| `unbounded_range` | Introduced from version 0 or defaultStatus=affected | Likely affects all versions, verify manually |
| `unmapped_cpe` | NVD CPE too broad to extract versions | Skip or flag for review |

### 5. Multi-Source Correlation

vuln-ingest stores affected_packages from each source separately (the `source` field). The matching service sees all perspectives:

```json
{
  "affected": [
    {
      "source": "nvd",
      "vendor": "xmlsoft",
      "package_name": "libxml2",
      "version_ranges": [{"introduced": "2.9.0", "fixed": "2.12.6"}]
    },
    {
      "source": "cvelistv5",
      "vendor": "xmlsoft",
      "package_name": "libxml2",
      "version_ranges": [{"introduced": "2.9.0", "fixed": "2.12.6", "range_type": "semver"}]
    },
    {
      "source": "ghsa",
      "package_name": "libxml2",
      "version_ranges": [{"introduced": "2.9.4", "fixed": "2.12.6", "range_type": "ECOSYSTEM"}],
      "quality_flags": []
    }
  ]
}
```

**Best strategy:** If any source with clean data (no quality flags) says the version is affected, treat it as affected. Use the most specific range available — prefer sources with `range_type` set over those without.

### 6. Enrichment Data

After matching, the matching service can use enrichment data to prioritize:

- **EPSS score** (`epss` field on vulnerability) — exploit probability. Higher = more urgent.
- **KEV status** (`kev` field) — CISA known-exploited. If present, the CVE is actively exploited in the wild and has a remediation deadline.

Use `?enrich=false` on bulk queries when you only need version ranges, then selectively enrich matched CVEs.

## Change Detection

The matching service needs to re-check when new CVEs appear or existing ones are updated.

**Poll the change feed:**
```
GET /v1/vulns?updated_since=2026-05-17T00:00:00Z&enrich=false&limit=1000
```

`updated_since` uses the local write timestamp (not the source's `modified` field), so it catches corrections where the source data changed but the source didn't bump its modified date.

**Recommended cadence:** Poll every 5–15 minutes. The syncer CronJob enqueues source checks on the same interval.

## Agent-Assisted Matching

AI agents reduce manual effort at four points in the matching workflow. In each case, the agent does the research and proposes an answer — the engineer reviews and approves.

### Auto-populate `cve_identifiers`

The 440 identity mappings are the setup bottleneck. An agent can:

1. Read the RPM spec file — extract `Name:`, `Version:`, `URL:`, `Source0:`
2. Query NVD's CPE dictionary for matching vendor/product pairs
3. Query cvelistV5 for vendor/product references to the same project
4. Cross-reference: if both NVD and cvelistV5 agree on `xmlsoft/libxml2`, high confidence
5. Write the `cve_identifiers` block to the metadata file

Agent proposes all 440 mappings. Engineer bulk-reviews. Most are obvious (`curl` → `haxx/curl`). Agent flags ambiguous cases (multiple CPE matches, name mismatches) for closer review.

### Resolve ambiguous version comparisons

Not all versions map cleanly:

| RPM version | Problem | Agent action |
|-------------|---------|-------------|
| `kernel-6.12.0-0.rc7.20241104git9f0c7d36bd07.54.hb1` | Git snapshot embedded in NVR | Parse commit hash, check if fix commit is ancestor |
| `glibc-2.40.9000` | Development snapshot | Find nearest upstream release tag, determine if fix included |
| `python3.12-3.12.8` | PEP 440 versioning | Apply ecosystem-specific comparison |
| `openssl-1:3.2.1-2.hb1` | RPM epoch | Strip epoch + dist tag, compare upstream version |

The agent parses the NVR, determines the upstream version, selects the correct comparison algorithm based on `range_type`, and reports the match result with reasoning.

### Cross-source conflict resolution

When sources disagree on affected ranges:
```
NVD:       introduced 2.9.0,  fixed 2.12.6
cvelistV5: introduced 2.9.4,  fixed 2.12.6
GHSA:      introduced 2.10.0, fixed 2.12.6
```

The matching service uses the broadest range for safety. The agent narrows it by reading the CVE description, finding the commit that introduced the vulnerability, and determining which `introduced` version is correct. It reports: "NVD says 2.9.0 but the vulnerable function was added in commit abc123, first appearing in 2.9.4 — cvelistV5 is correct."

### Unmatchable CVE recovery

CVEs with `empty_range` or `unmapped_cpe` flags have no version data — the matching service can't process them automatically. An agent can:

1. Read the CVE description to identify the affected project
2. Find the fix commit in the upstream repo
3. Determine the affected version range from git history
4. Propose the range to the advisory database

This recovers CVEs that would otherwise wait weeks/months for NVD to add CPE data.

## Automation Summary

| Step | Automated? | How |
|------|-----------|-----|
| RPM identity mapping | Agent-proposed, engineer-reviewed | Agent reads spec + queries CPE → proposes `cve_identifiers` |
| Bundled dependency mapping | Fully automated | Konflux SBOM → PURLs |
| CVE data ingestion | Fully automated | vuln-ingest fetcher + resolver |
| Vulnerability matching | Fully automated | Batch query + version comparison |
| Ambiguous versions | Agent-assisted | Agent parses NVR, selects comparator, reports reasoning |
| Source conflicts | Agent-assisted | Agent reads advisories + commits, recommends correct range |
| Unmatchable CVEs | Agent-assisted | Agent reads description + git history, proposes range |
| Quality flag triage | Agent-assisted | Agent drafts VEX justification, engineer approves |
| Enrichment prioritization | Fully automated | EPSS score + KEV status |
| Change detection | Fully automated | `updated_since` polling |

The agent pattern throughout: **do the research, propose the answer, engineer reviews.** The research — reading advisories, checking versions, cross-referencing sources — is where 80% of the time goes and where agents add the most value.
